### Post-mortem

Written at sprint close (2026-06-08).

**What went well?**

- **ADR-driven design upfront paid off.** ADR-0008 enumerated three options and settled on Option B (three packages) before any code was written. The Abstractions extraction became mechanical once the boundary was clear — no rework.
- **Abstractions extraction was low-risk.** Moving `ILogShipper`, `BrokerLogEvent`, `IRequestHandler`, and `RequestResult` into the new `ElsaBroker.Abstractions` project was a refactor with a well-defined surface. All 16/16 tests continued to pass with 0 warnings after the move.
- **Parallel agent execution compressed the schedule.** dotnet-architect owned the project + pack + Dockerfiles while docs-web authored the deployment quickstart simultaneously. The L card was delivered in a single session — consistent with v0.2.0 velocity.
- **`dotnet pack` validation was immediate.** Producing and inspecting the three `.nupkg` files (`ElsaBroker.Contracts.0.3.0.nupkg`, `ElsaBroker.Abstractions.0.3.0.nupkg`, `ElsaBroker.0.3.0.nupkg`) confirmed the package graph was correct before any docs were finalized.
- **Solution file gap caught and closed.** The missing `Release|Any CPU` build configuration mappings were discovered during packaging and fixed as part of this sprint — a latent defect that would have surfaced at any CI/CD step.

**What could improve?**

- **Solution file debt from initial scaffolding.** The `Release|Any CPU` configuration mappings were missing from the original scaffold. This should have been caught in the v0.1.0 sprint when the solution file was first verified. Going forward: validate both `Debug` and `Release` build targets when a project is first added to the solution.
- **`.dockerignore` coordination gap.** When `ElsaBroker.Queue/Dockerfile` and `ElsaBroker.Processor/Dockerfile` were added, the `.dockerignore` needed updating because a prior per-project exclusion would have broken the shared build context. This is a predictable coordination issue any time a new Dockerfile is introduced; the dotnet-architect should add `.dockerignore` review to the Dockerfile checklist.

**Process changes adopted**

- _(None required this sprint. The v0.1.0/v0.2.0 process changes are holding. The two improvement items above are team reminders, not process-document changes — they are captured in the release report for the next sprint's reference.)_
