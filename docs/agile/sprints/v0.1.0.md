### Post-mortem

Written at sprint close, 2026-06-08. Each agent that owned a card contributes a bullet.

**What went well?**

- (project-manager) Sprint process worked end-to-end: backlog groomed, cards tracked, board
  checkpointed after every move. Kanban survived a multi-step session without data loss.
- (dotnet-architect) ADR-0007 research was thorough — web search for Elsa's MassTransit integration
  combined with deep source reads produced a well-grounded recommendation. Parallel agent launch
  (Explore for research, then dotnet-architect for writing) was efficient.
- (docs-web) EB-005 and EB-006 ran in true parallel (different file surfaces: Docker vs docs). No
  merge conflicts, no coordination overhead. The three integration gotchas capture is valuable —
  saves future debugging.
- (project-manager) Gating mechanism (EB-001 gates EB-002/003/004) worked as designed — the ADR
  landed before downstream work started, preventing rework.

**What could improve?**

- (project-manager) The v0.1.0 release report was written before the sprint's feature work landed
  (EB-000 cut the release early). The close step now needs to amend the report — a minor friction.
  Consider deferring the release report to sprint close rather than cutting it mid-sprint.
- (dotnet-architect) ADR numbering: the backlog acceptance criteria said "0008+" but the next
  available slot was 0007. Minor — but the PM should verify numbering at grooming time, not leave it
  to the author.
- (docs-web) The Elsa Studio image's internal port assumption (80 vs 8080) caused a small doc error
  that needed a manual fix. Compose config should be the single source of truth for port numbers.

**Process changes adopted**

- Post-mortem step added to the sprint lifecycle (this section) — applied to `docs/agile/README.md`
  and `sprints/_template.md`. Runs automatically at sprint close, before proposing the next sprint.
- Close → post-mortem → next sprint is now a single automatic sequence (codified in the process README).
